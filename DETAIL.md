# NSG 포털 그룹웨어 알림 시스템 분석 보고서

본 문서는 `nsgportal1` 프로젝트의 소스 코드 구조, 기능, 동작 프로세스 및 테스트 체계를 상세 분석한 내용입니다.

---

## 1. 프로젝트 개요
NSG 포털 그룹웨어의 새 게시물과 오늘의 식단 정보를 크롤링하여 **카카오톡(나에게 보내기)** 및 **디스코드(웹훅)**로 전송해 주는 파이썬 기반의 자동화 모니터링 시스템입니다.
* **주요 목적**: 사내 주요 게시판 모니터링 자동화 및 매일 제공되는 식단 정보 전달의 편의성 제공.
* **주요 기능**:
  - 새 게시물 실시간 감지 및 수정된 게시물 알림
  - 평일 오전 10시 식단 정보(중식/석식 및 칼로리) 전송
  - 카카오톡 OAuth 토큰 자동 갱신 및 secrets.json 원자적(Atomic) 업데이트
  - 디스코드 웹훅 연동을 통한 다중 채널 알림
  - PID 기반 lockfile을 활용한 중복 실행 방지

## 3. 핵심 모듈별 상세 분석

### ① 진입점 제어 (`groupware_notifier/main.py`)
* **역할**: 모니터링 태스크 실행 주기 제어, 중복 실행 방지 및 예외 처리.
* **주요 기법**:
  * **PID 기반 중복 실행 방지 (`acquire_lock`, `release_lock`)**:
    - `run.lock` 파일에 현재 프로세스 ID(PID)를 기록하여 다중 실행을 방지합니다.
    - 만약 비정상 종료 등으로 락 파일이 남아있는 경우(Stale lockfile), `psutil.pid_exists(pid)`를 사용하여 실제 프로세스가 살아있는지 검사하고, 죽어있다면 락 파일을 지우고 정상 실행하도록 하여 장애 내구성을 높였습니다.
  * **동작 모드 구분**:
    - 기본 실행: 게시판의 새 글 확인 및 수정을 모니터링합니다. 첫 실행 시에는 알림을 보내지 않고 현재 최신글을 기준으로 `state.json`에 저장(기준점 설정)합니다.
    - `--meal` 옵션 실행: 오늘의 중식/석식 식단 및 칼로리를 조회하여 전송합니다.
  * **수정 게시물 감지**:
    - 이전 루프에서 수집한 타이틀 목록(`seen_titles`)과 새로 수집한 타이틀을 비교하여 동일 ID에서 제목이 변경된 경우 `✏️ 게시물 수정` 알림을 보냅니다.

### ② 그룹웨어 스크래퍼 (`groupware_notifier/scraper.py`)
* **역할**: SiteMinder SSO 로그인 처리 및 HTML 파싱을 통한 게시글/식단 추출.
* **주요 기법**:
  * **SiteMinder SSO 로그인 흐름**:
    1. `entry_url`을 호출하여 `ssoTarget` 정보를 파싱합니다.
    2. 비밀번호를 SHA-256 해시로 변환합니다.
    3. `company_id`(예: `CD` -> `D`)의 접두사와 `username`(사번)을 합쳐 ID를 생성합니다.
    4. `login_url`로 로그인 폼 데이터를 POST 요청한 후 `ssoLogin.jsp`로 연계 POST를 호출하여 최종 포털 세션 쿠키(`JSESSIONID`)를 얻어냅니다.
  * **세션 만료 자동 감지 및 재시도 (`_get_with_session_check`)**:
    - 게시판 데이터 GET 요청 시 리다이렉트되어 최종 URL에 `login_url`이 포함되거나, 응답 HTML 내에 `<input type="password">` 태그가 발견되면 세션이 만료된 것으로 판단합니다.
    - 세션 만료 시 즉시 1회 자동 재로그인(`login()`)을 수행한 후 기존 요청을 재시도합니다.
  * **게시판 파싱**:
    - BeautifulSoup을 사용해 설정된 `post_selector`(기본 `tr`) 내에서 `post_id_pattern`(fnViewAtcl 정규식) 또는 `post_id_attr` 속성을 바탕으로 글 ID를 추출합니다.
  * **식단표 파싱**:
    - 날짜 테이블에서 오늘 날짜(`%Y.%m.%d`)와 일치하는 열 인덱스(`col_idx`)를 구한 뒤, 식단 상세 테이블의 해당 열 데이터를 파싱하여 중식/칼로리/석식 메뉴 정보를 추출합니다.

### ③ 알림 엔진 (`groupware_notifier/notifier.py`)
* **역할**: 카카오톡 및 디스코드로 알림 전송 및 OAuth 토큰 관리.
* **주요 기법**:
  * **KakaoNotifier**:
    - 카카오톡 "나에게 보내기" API (`/v2/api/talk/memo/default/send`)를 사용합니다.
    - 401 Unauthorized 에러 응답 수신 시, `secrets.json`에 저장된 `refresh_token`을 활용해 토큰 갱신 API (`/oauth/token`)를 호출하여 새 토큰으로 자동 갱신하고 재시도합니다.
  * **DiscordNotifier**:
    - 디스코드 웹훅 주소로 Embed 양식의 메시지를 JSON 바디로 구성하여 POST 전송합니다.
  * **MultiNotifier 및 팩토리 함수**:
    - `config.json`의 `notifiers` 필드 정보(`kakao`, `discord`)에 맞춰 가용한 알림 모듈들을 활성화한 후 리스트로 관리합니다.
    - 알림 발송 중 특정 노티파이어가 실패하더라도 에러 로그만 기록하고 다른 노티파이어 전송이 끊기지 않도록 처리되어 있습니다.
  * **원자적 파일 쓰기 (`_write_json_atomic`)**:
    - 파일 쓰기 도중 프로세스가 비정상 종료되어 JSON 파일이 깨지는 현상을 방지하기 위해 임시 파일(`.tmp`)에 먼저 저장한 후 `os.replace()`를 호출해 덮어씁니다.

### ④ 카카오 토큰 초기 설정 스크립트 (`groupware_notifier/setup_token.py`)
* **역할**: 최초 1회 수동 실행하여 카카오 API 연동에 필요한 Access Token 및 Refresh Token을 획득합니다.
* **동작 흐름**:
  1. `webbrowser` 모듈을 사용해 사용자의 PC에 카카오 로그인 페이지를 띄웁니다.
  2. 사용자가 로그인 및 권한(talk_message) 동의를 마친 뒤 리다이렉트된 주소를 터미널에 붙여넣습니다.
  3. URL 쿼리 파라미터에서 `code`(인증 코드)를 추출하여 `authorization_code` 방식으로 토큰을 교환한 뒤 `secrets.json`에 저장합니다.

## 4. 테스트 체계 (`tests/` 디렉토리)
프로젝트는 `pytest` 및 `responses` 라이브러리를 사용하여 오프라인 환경에서도 스크래핑과 알림 발송 로직을 검증할 수 있는 단위 테스트 코드가 완벽히 갖춰져 있습니다.

* **`test_main.py`**:
  - `state.json` 정상 로드 및 원자적 쓰기 검증
  - 중복 프로세스 진입 시 락 차단 테스트 및 이미 존재하지 않는 PID의 stale lock 청소 확인
  - 첫 실행 시 알림 없이 상태 기준점만 생성하는 흐름 테스트
  - 새 게시물이 있는 경우 알림 발생 및 `last_seen_id` 갱신 여부 테스트
* **`test_notifier.py`**:
  - 카카오 메시지 전송 시 401 에러를 받으면 모킹된 토큰 갱신 흐름을 거쳐 새 토큰으로 자동 재발송하는 지 검증
  - 토큰 갱신 에러 발생 시 `RuntimeError`가 유발되는지 확인
  - 갱신 응답에 `refresh_token`이 없을 때 기존 토큰 정보 유실 방지 테스트
* **`test_scraper.py`**:
  - 로그인 크레덴셜 거부 등으로 로그인 실패 시 `RuntimeError` 발생 확인
  - 세션 만료 응답(302 혹은 로그인 폼 반환) 시 자동 재로그인을 거쳐 데이터 조회에 성공하는 시나리오 테스트
  - 파싱 행 수 0개일 때 빈 리스트 반환 및 적절한 경고 로깅 테스트

## 5. 향후 과제 및 개선 계획 (TODOS.md 참고)
* **장애 감지 기능 (Dead man's switch)**:
  - 매일 특정 시간(예: 오전 9시)에 카카오톡으로 "알림 정상 동작 중" 하트비트 메시지를 발송하여 스크립트의 작동 중단 여부를 간접 모니터링할 수 있는 장치 마련이 필요합니다.
* **Rate Limit 모니터링**:
  - 5분 간격의 주기적인 세션 요청 및 재로그인이 그룹웨어 서버 측에 세션 잠금이나 제한을 유발하는지 실서버 운영 중 모니터링이 필요합니다.
